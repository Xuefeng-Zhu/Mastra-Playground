- Impact: Malformed tool inputs fail at runtime with cryptic errors instead of compile-time type errors.

### [HIGH] Empty catch blocks swallow errors silently

- File: `shared/traced-step.ts:76-78`
- Category: Error handling
- Description: The `timed()` wrapper catches errors but only extracts the message string; the original error object and stack trace are lost.

### [HIGH] Wrong attribute selector breaks HITL suspended-node highlighting

- File: `public/app.js:1191-1198`
- Category: Correctness bug
- Description: `renderGraph()` sets `data-node` on SVG `<g>` elements, but the HITL override functions query `data-node-id`, which doesn't exist. The `.suspended` and `.active` classes are never applied to the gate/execute nodes.
- Evidence:


- Impact: When a workflow suspends in HITL mode, the gate node in the graph UI never gets the orange "suspended" highlight — the user gets no visual cue.
- Fix: Change `data-node-id` to `data-node` in both override function queries (lines 1192-1198).

### [MEDIUM] `result.error` does not exist on WorkflowResult — errors swallowed as `[object Object]`

- File: `examples/01-support-triage/index.ts:182`
- Category: Error handling / Mastra API misuse
- Description: `result.status !== 'success'` implies an error exists, but `WorkflowResult` doesn't have an `.error` property. The fallback `String(result)` renders `[object Object]` for object errors, making failure diagnostics useless.
- Evidence:

```typescript
error: result.status !== 'success' ? (result as { error?: string }).error ?? String(result) : null,
```

- Impact: When a workflow fails, the UI shows `[object Object]` or `undefined` instead of the real error message.
- Fix: Use `result.status === 'failed' ? String(result) : null` or access the correct Mastra error field.

### [MEDIUM] Same `result.error` issue in examples 02, 03, 04, 05

- File: `examples/02-research-agent/index.ts:105`, `examples/03-code-review-agent/index.ts:204`, `examples/04-parallel-research/index.ts:220`, `examples/05-multi-turn-chat/index.ts:224`
- Category: Error handling
- Description: All four examples use `String(result)` as error fallback, which produces `[object Object]` for non-string errors. Additionally, none of them access the actual Mastra error field.
- Evidence:

```typescript
error: result.status !== 'success' ? String(result) : null,
```

- Impact: Identical to the above — failed workflow runs show opaque `[object Object]` errors.
- Fix: Replace with `result.status !== 'success' ? (result.result ? String(result.result) : String(result)) : null` or use Mastra's actual error field.

### [MEDIUM] HITL graph is missing edges for rejected and auto-approved paths

- File: `public/app.js:166-178`
- Category: UX issue
- Description: The HITL graph definition has only one edge (`input → classify → gate → execute`) with a conditional `when: { kind: 'auto-approved' }`. There are no edges showing the `rejected` path or the `approved` human path.
- Evidence:

```javascript
'hitl-approval': {
  nodes: [/* input, classify, gate, execute */],
  edges: [
    { from: 'input', to: 'classify' },
    { from: 'classify', to: 'gate' },
    { from: 'gate', to: 'execute', label: 'approved', when: { kind: 'auto-approved' } },
  ],
},
```

- Impact: The graph doesn't accurately represent the HITL branching. When a human approves, no edge is shown. When rejected, no edge is shown. Misleading for learners.
- Fix: Add two edges from `gate` to `execute` with labels "approved (human)" and "rejected", and add `when` conditions or predicates to show the auto-approved edge distinctly.

### [MEDIUM] Settings persistence swallows JSON.parse errors silently

- File: `public/app.js:50-54`
- Category: Error handling
- Description: Both `settingsStore.load()` and `settingsStore.loadFor()` wrap `JSON.parse` in try/catch and return `{}` or `[]` on failure. If localStorage is corrupted, the user loses settings with no indication.
- Evidence:

```javascript

  const raw = localStorage.getItem(SETTINGS_KEY);
  const all = raw ? JSON.parse(raw) : {};
  return all[example] || {};
} catch {
  return {};
}
```

- Impact: Settings silently revert to defaults; user has no idea their preferences were lost.
- Fix: Log a console.warn when parse fails and consider providing a "reset settings" affordance.

### [INFO] Agent instances are created fresh per step execution in all examples

- File: `examples/01-support-triage/index.ts:46-58`, `examples/03-code-review-agent/index.ts:94-104`, etc.
- Category: Mastra API misuse / inefficiency
- Description: Every `createStep` `execute` callback builds `new Agent(...)` from scratch. Mastra's own documentation recommends reusing agent instances. Creating agents per step is slow and may cause connection pool issues.
- Impact: Performance degradation on each step execution; agent initialization overhead on every run.
- Fix: Build agents once outside the step factory and close over them, or use Mastra's agent registry.

### [INFO] HITL `captureRun` is called AFTER `createRun()` but BEFORE `start()`

- File: `examples/06-hitl-approval/index.ts:275-278`
- Category: Race condition risk
- Description: `captureRun(run)` is invoked between `createRun()` and `run.start()`. If the workflow suspends synchronously during `start()`, the gate step's `execute` tries to access `capturedRun` before `captureRun` has been called. In practice, suspension is async, but this is a fragile ordering.
- Evidence:

```typescript
const run = await wf.createRun();
built.captureRun(run); // registered here
built.captureMastra(built.mastra);
const result = await run.start({ inputData: { action: input.action, actionType: input.actionType } });
// suspend could happen inside start() above
```

- Impact: If Mastra ever suspends synchronously in `start()`, `capturedRun` would be null and `registerSuspendedRun` would silently skip registration, losing the token.
- Fix: Capture the run object before calling `start()`, or document the async-only guarantee from Mastra.
